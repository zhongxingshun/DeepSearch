### DQ-001: 缺少定时备份任务设计

- **评审意见**: 设计文档缺少备份任务的具体设计，无法满足 SRS 3.2 中 RPO ≤ 1小时的要求
- **建议修改**: 需补充 Celery Beat 定时任务配置，实现 PostgreSQL、Meilisearch、配置文件的定时备份
- **答复**: ✅ 接受
- **答复理由**: RPO 是关键的非功能需求，必须在设计阶段明确备份机制
- **修改内容**: 
  - 新增第 7.4 节"定时任务设计"
  - 增加 Celery Beat 配置，定义 4 个定时任务：
    - PostgreSQL 增量备份 (每小时)
    - PostgreSQL 全量备份 (每日 02:00)
    - Meilisearch 快照 (每日 03:00)
    - 历史记录清理 (每日 04:00)
  - 新增 backup_task.py 任务模块
- **对应章节**: TDD v1.1 第 7.4 节

### DQ-002: 缺少日志清理机制

- **评审意见**: audit_logs 表会持续增长，需要清理机制满足 SRS 3.3 日志保留 180 天的要求
- **建议修改**: 需补充 audit_logs 表的定期清理策略
- **答复**: ✅ 接受
- **答复理由**: 日志自动清理是保障系统稳定运行的必要机制
- **修改内容**: 
  - 在第 7.4 节定时任务中增加"审计日志清理"任务
  - 清理策略：删除 created_at < 180 天前的记录
  - 搜索历史清理：删除 created_at < 90 天前的记录
- **对应章节**: TDD v1.1 第 7.4 节

### DQ-003: 缓存层设计不够详细

- **评审意见**: 性能设计部分缺少 Redis 缓存的具体使用场景
- **建议修改**: 补充热门搜索结果缓存、用户会话缓存、接口限流计数器
- **答复**: ✅ 接受
- **答复理由**: 缓存是提升系统性能的关键手段，应在设计阶段明确
- **修改内容**: 
  - 新增第 7.5 节"缓存设计"
  - 定义 4 种缓存场景：
    - 会话缓存 (TTL: 24h)
    - 限流计数器 (TTL: 1min)
    - 用户信息缓存 (TTL: 5min)
    - 系统配置缓存 (TTL: 10min)
  - 明确不缓存搜索结果（Meilisearch 已足够快）
- **对应章节**: TDD v1.1 第 7.5 节

### DQ-004: 文件存储目录结构未详细定义

- **评审意见**: /data/files 目录结构未定义，可能导致单目录文件过多
- **建议修改**: 建议按日期或 hash 分桶
- **答复**: ✅ 接受
- **答复理由**: 合理的目录结构对文件管理和性能都有重要影响
- **修改内容**: 
  - 修改第 10.1 节"目录结构"，补充 data/files 子目录设计
  - 采用 MD5 前两位分桶策略：/data/files/{md5[0:2]}/{md5}.{ext}
  - 优点：分布均匀（256 个子目录），便于定位
- **对应章节**: TDD v1.1 第 10.1 节

### DQ-005: 健康检查接口未定义

- **建议修改**: 增加 /api/v1/health 接口
- **答复**: ✅ 接受
- **答复理由**: 健康检查是容器化部署的标准实践
- **修改内容**: 
  - 在第 3.2 节模块接口定义中增加健康检查接口
  - 接口路径: GET /api/v1/health (无需认证)
  - 检查项：数据库连接、Redis 连接、Meilisearch 连接
  - 响应格式：{ status: "healthy/unhealthy", components: {...} }
- **对应章节**: TDD v1.1 第 3.2.6 节
